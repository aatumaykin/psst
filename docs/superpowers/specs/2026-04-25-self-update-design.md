# Self-Update Design

## Goal

Add `psst update` and `psst update check` commands to check for new versions and perform in-place binary self-update from GitHub Releases.

## Commands

```
psst update check       # Check if a newer version exists
psst update             # Download and install the latest version
psst update --force     # Install even if the version is the same
```

Global flags supported: `--json`, `--quiet`.

## Source

GitHub Releases API (public, no token required):

- `GET https://api.github.com/repos/aatumaykin/psst/releases/latest`
- Response field `tag_name` contains the semver tag (e.g. `v1.3.0`)
- Assets include `psst_{version}_{os}_{arch}.tar.gz` (`.zip` for Windows) and `checksums.txt`

## Architecture

New package: `internal/updater` — a leaf package depending only on stdlib and `internal/version`.

### File structure

```
internal/updater/
  updater.go    — types, version comparison, orchestration
  github.go     — GitHub Releases API client
  verify.go     — SHA256 checksum verification
  install.go    — download, extract, atomic binary replacement

internal/cli/
  update.go     — Cobra command wiring (subcommand of root)
```

### Dependency graph

```
cli/update.go → internal/updater → internal/version
                                    stdlib (net/http, archive/tar, compress/gzip, crypto/sha256, os, runtime)
```

No dependencies on vault, store, crypto, keyring, runner, or output (updater formats its own output).

## Data Flow

### `psst update check`

1. GET `/repos/aatumaykin/psst/releases/latest`
2. Parse `tag_name` as semver
3. Compare with `version.Version` using semver comparison
4. Output: current vs latest, or "Already up to date"

### `psst update`

1. Same check as `update check`
2. If already up to date and not `--force` — print message and exit 0
3. Download `checksums.txt` from release assets into temp directory
4. Download platform archive (`psst_{version}_{os}_{arch}.tar.gz` or `.zip`) into temp directory
5. Parse `checksums.txt` to find expected SHA256 for the archive
6. Verify downloaded archive SHA256 matches
7. Extract binary from archive
8. Atomic replacement:
   - `os.Rename(currentBinary, currentBinary + ".bak")`
   - `os.Rename(newBinary, currentBinary)`
   - Remove `.bak` on success
9. Clean up temp directory

## OS/Arch Detection

- `runtime.GOOS` and `runtime.GOARCH` determine the archive name
- Template: `psst_{version}_{os}_{arch}.tar.gz` (Windows: `.zip`)
- goreleaser already produces these archives via `.goreleaser.yml`

## Error Handling

| Scenario | Behavior |
|----------|----------|
| Network error | "Unable to check for updates: <error>" |
| GitHub rate limit (403) | "GitHub API rate limit exceeded. Try again later." |
| Checksum mismatch | Delete downloaded files, "Checksum verification failed. Download may be corrupted." |
| Permission denied | "Permission denied. Try running with elevated privileges." |
| Cross-device rename | Fallback: copy new binary, then delete old |
| Current binary is `dev` build | Allow update (dev < any real version) |

## Semver Comparison

Manual implementation without external dependencies:

1. Strip `v` prefix from both versions
2. Split on `.` into major/minor/patch
3. Compare numerically: major first, then minor, then patch
4. Pre-release tags (e.g. `-beta`) are treated as older than the release version

## Verification

SHA256 checksum from `checksums.txt` (generated by goreleaser):

1. Download `checksums.txt`
2. Parse line format: `<hash>  <filename>`
3. Find line matching the downloaded archive filename
4. Compute SHA256 of downloaded archive
5. Compare

## Testing

### Unit tests (`internal/updater/`)

- `TestCompareVersions` — semver comparison edge cases
- `TestParseChecksums` — parse checksums.txt format
- `TestBuildAssetName` — correct archive name for each OS/arch
- `TestCheckUpdate` — mocked GitHub API via `httptest.Server`

### Integration tests

- Download + verify + extract using a real tar.gz in temp directory
- Atomic replacement test with temp binaries

### Test constraints

- No real HTTP calls to GitHub in tests — use `httptest.Server`
- No real binary replacement — use temp directories
- Follow existing test patterns in the project

## Security Considerations

- No secrets involved in update process
- SHA256 verification prevents tampering
- HTTPS-only for all downloads
- No auto-update — user explicitly runs the command
- Backup of old binary allows rollback if rename fails mid-operation
